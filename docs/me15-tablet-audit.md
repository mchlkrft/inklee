# ME-15 iPad/tablet audit — findings, decisions, and implementation spec

Produced 2026-07-17 by a 20-agent audit workflow (13 area readers over every route file, 3 cross-cutting sweeps, 3 strategy designers, 1 adversarial completeness critic; ~1.24M tokens). This document is the Phase 1 deliverable of ME-15 (roadmap §6.4, launch-gate blocker 10) and the binding spec for Phases 2–5. Raw per-agent JSON lives in the session scratchpad; the deduped per-screen inventory and sweep findings are appended at the bottom.

## 1. Discovered architecture (verified, not assumed)

- Expo SDK 54.0.35, RN 0.81.5, expo-router 6.0.24, @react-navigation/native 7.2.5, bottom-tabs 7.16.2, react-native-screens 4.16.0, reanimated 4.1.7, nativewind 4.2.5, safe-area-context 5.6.2. No expo-updates (no OTA: every change ships via EAS build). No expo-screen-orientation.
- 57 routable screens + 8 layouts. `src/components/Screen.tsx` (SafeAreaView + `px-5` box, `padded`/`topBar` props, **no max-width**) wraps ~47; off-contract screens: bookings/[id], clients/[email], insights, waitlist ×2, account/delete, notifications, onboarding/index, auth-confirm.
- Navigation chrome is fully custom: `BottomNav.tsx` floating pill (absolute overlay, center FAB, keyboard-hide + detail-route-hide logic, exports `TAB_BAR_CLEARANCE=120` consumed at 9 sites + 2 literal `120`s; 6 nested detail screens hardcode `paddingBottom:48` deliberately) wired via the `tabBar` render prop; `TopBar.tsx` floating overlay pill with reanimated scroll-hide (`scroll-hide.ts` module-level shared `makeMutable`).
- **Load-bearing discovery:** installed bottom-tabs 7.16.2 supports `tabBarPosition: 'left'` AND still calls the custom `tabBar` render prop (`BottomTabView.tsx:242-253`: root goes `flexDirection:'row'`, tab bar renders before the `flex:1` screens container). A non-absolute rail therefore **structurally reserves its column** — screens, absolute overlays, and pinned bands all confine themselves to the content pane with zero per-screen left-inset work.
- Modals: two custom patterns on bare RN `<Modal>` (TopSheet ×2 consumers; bottom sheets ×4 flash + DateRangeField calendar + travel/map trips modal), all full-width, none set `supportedOrientations` (iOS default is portrait-only — a hard bug once rotation unlocks). All correctly re-apply `themeVars` (house rule) and pass `onRequestClose`.
- Data layer: TanStack Query v5 keyed `["api", path]` (30s staleTime) — a detail pane and a pushed detail route fetching the same path share one cache entry automatically; `invalidateBookingViews` fans out by path prefix.
- Grids today: goods index + flash days [id] (`numColumns=2`), flash/instagram (3-col via `useWindowDimensions` math), ReferenceImageGallery (31% wrap). Only 2 files read window width; no `Dimensions.get` anywhere; no breakpoint helper exists.
- Sweep verdicts: scroll structure healthy (zero same-axis nested scrollables); a11y strongest at the shared-component layer (roles/labels/states present), weakest at sheet focus management, hardware-keyboard Escape, and returnKeyType chains.

## 3. The adaptive system (implementation spec)

### 3a. `src/lib/layout.tsx` (new)
`LayoutClass`, `BREAKPOINT = { medium: 600, expanded: 900 }`, `classify(w, h)`, `WindowLayoutProvider` (mounted in root `_layout.tsx` inside ThemeProvider), `useLayoutClass()`, `useIsExpanded()`, `useWindowClassTransition(cb)` (fires once per class flip via prev-ref), `useTabBarClearance()`, `useScreenGutter()` (20/24/32), `CAP`/`PANE`/`SHEET_MAX` constants, `useGridColumns({ minTile, gap, max, width })` returning `{ numColumns, tileWidth, key }`.

### 3b. `Screen` gains `column?: "form" | "feed" | "wide"`
Inner `View style={{ maxWidth: CAP[column], width: "100%", alignSelf: "center" }}` when set; gutter from `useScreenGutter()`. Adoption is a one-word diff for ~47 screens; the 8 off-contract screens get individual treatment in Phase 4.

### 3c. Navigation (per the nav strategy, adopted whole)
`src/lib/nav-items.ts` (ICONS/LABELS extracted from BottomNav); new `src/components/NavRail.tsx` (96pt fixed-width chrome-shell rail: brand mark, 5 destinations with the Bookings FAB identity kept, bottom cluster = BooksStatusPill compact variant + NotificationBell + account menu; no keyboard-hide, no detail-hide, never null); `(tabs)/_layout.tsx` flips `tabBarPosition` + tabBar branch on `expanded`; `TopBar` returns null at expanded and `useTopBarHeight()` shrinks to `insets.top+12` (the bookings pinned band and all content tops adapt automatically); BottomNav + TopBar pills get `maxWidth` caps at medium; `scroll-hide.ts` short-circuits and pins progress to 0 at expanded.

### 3d. Sheets
New `AdaptiveSheet` (extracted shared skeleton): compact = today's edge-anchored bottom sheet; medium/expanded = centered card dialog (`SHEET_MAX`, maxHeight 90%, all corners rounded) over the same backdrop. Bakes in: `themeVars` root (house rule), `onRequestClose`, `supportedOrientations`, a `KeyboardAvoidingView` slot (ManageFolderSheet is the test case). Migrate: FlashItemSheet, NewDesignSheet, AddToFolderSheet, ManageFolderSheet, DateRangeField's calendar modal, travel/map trips modal. TopSheet keeps top anchor but caps its panel ~400pt pinned top-right; gets `supportedOrientations` too.

### 3e. Master-detail (Phase 3)
`ListDetailHost` (`flex-row`; list pane 380 + hairline + detail pane with inner `CAP.detailContent` column; `DetailPaneContext { embedded, close }`; `key={selected}` remount rule) + content extractions `BookingDetailContent` / `ClientDetailContent` (mechanical moves — both are pure functions of one param with their own fetch) + thin route wrappers implementing Rule A + `selected` search param on list routes (the `tripId` param idiom proves delivery) + Rule B transition hook + draft store. Calendar: same-screen `flex-row` conditional, no routing (its `selectedDate` state survives class flips for free). Known v1 concession: SubNav section switches (`router.replace`, no params) drop the selection — documented, acceptable.

### 3f. Grids
`useGridColumns` on: goods index (2 → 3-4 cols, replaces the 48.5% hack), flash/instagram (3 → 5-7 under the feed cap; pinned Import footer aligns to the cap), flash days [id] picker (tileWidth only), ReferenceImageGallery thumbnails. FlatList consumers spread `key` (numColumns remount rule) and accept scroll-position loss on mid-drag resizes.

## 4. Ranked risk register

1. **Orientation config correctness** (D2): wrong combination either fails App Store validation or kills Split View. Verify via `npx expo config` introspection + on-device rotation before submitting.
2. **Every missed `supportedOrientations` Modal snaps the iPad to portrait** — must be a sweep with a grep gate (`<Modal` census), not incremental drift.
3. **`tabBarPosition:'left'` + custom tabBar** is an uncommon combination: smoke-test scene width/freeze/RTL in the first build; fallback = keep bottom pill at expanded (rail deferred) with caps only.
4. **Live-resize edge cases**: class flips mid-mutation (pane remount drops busy spinners — self-heals via invalidation), FlatList remounts lose scroll position, a stranded hidden TopBar must unpin (progress reset effect covers it). Bias: accept cosmetic blips, never lose data (draft store).
5. **Bookings full-bleed `-20/+20` negative-margin band** silently misaligns under any cap/pane: rework the band edge handling when the requests list is capped or paned (`the host owns pane width; never re-pad the list`).
6. **Android tablets reach medium/expanded immediately** (12L+ ignores the portrait lock): the layout work ships to them untested-by-hardware; rely on the width system + Play pre-launch report.
7. **A11y debt amplifies with hardware keyboards** (sheet focus, Escape, returnKeyType chains): AdaptiveSheet fixes the sheet story in one place; field chains are Phase 4-5 polish.
8. **No OTA**: every iteration is an EAS build (~30 min + TestFlight processing). Batch phases into planned builds (see §6); never ship supportsTablet without the foundation in the same build.

## 5. Testing matrix (feasible with available hardware)

Founder's iPad covers, natively: full-screen portrait/landscape, Split View narrow (~320), half (~507), two-thirds (~678+), Stage Manager arbitrary resize, hardware-keyboard if paired, plus rotation-during-flow. Android phones cover compact. Android tablet: no hardware — rely on width-driven design + emulated checks post-launch. Per window class, walk: sign-in → dashboard → requests list/detail (+ accept/pass) → calendar month/agenda → client detail (notes dirty + rotate) → deposit request form (dirty + resize) → flash grid + sheet → settings form → sheet keyboard behavior → push-notification deep link into a detail route at expanded (Rule A) → shrink mid-selection (Rule B). Automated: unit tests for `classify`, `useGridColumns` math, and cap constants (vitest, `packages/shared`-style pure functions; RN component tests are not in the repo's harness).

## 6. Implementation sequence (EAS build checkpoints)

- **Build T1 (Phase 2 + native config, one build):** layout.tsx + provider; Screen `column` + gutter; nav-items + NavRail + chrome switch + clearance hook migration; AdaptiveSheet + 6 migrations + TopSheet cap + Modal orientation sweep; grid hook on the 4 grids; `supportsTablet` + `~ipad` orientations. Founder smoke test on iPad (rail, rotation, Split View, sheets).
- **Build T2 (Phase 3):** calendar two-pane; ListDetailHost + BookingDetailContent + Requests pane + Rules A/B + draft store; ClientDetailContent + Clients pane.
- **Build T3 (Phase 4 + 5):** off-contract screen sweep, remaining caps, form group polish, a11y fixes, dead-code cleanup, final QA pass + deliverables report.

Each build: mobile tsc + lint + lucide gate; commit per phase; headless iOS submit + Android preview build.

## 7. Completion report (Phase 5, 2026-07-17) — the 15 deliverables

1. **Audit summary**: §1–§2 above (20-agent audit, 12 recorded decisions).
2. **Architecture discovered**: §1.
3. **Responsive strategy**: §3 (window-class context, canonical caps, adaptive chrome, state-driven master-detail).
4. **Layout classes**: compact <600 ≤ medium <900 ≤ expanded (+shortest-side ≥600 guard); rationale in D1.
5. **Screens reviewed**: all 57 routes + 8 layouts (Appendix A).
6. **Screens materially changed**: ~50 (chrome switch touches every tab screen; caps on ~40; master-detail on requests/clients/calendar; sheets on 10 modal sites).
7. **New shared primitives**: `layout.tsx` (WindowLayoutProvider, useLayoutClass, useIsExpanded, useWindowClassTransition, useTabBarClearance, useScreenGutter, gridColumns, CAP/PANE/SHEET_MAX), `Screen.column`, `NavRail`, `AdaptiveSheet`, `ListDetailHost`+`DetailPaneContext`, `draft-store`, `nav-items`, `BookingDetailContent`, `ClientDetailContent`.
8. **Native config changes**: `ios.supportsTablet: true`, `UISupportedInterfaceOrientations~ipad` all four (phones stay portrait), no `UIRequiresFullScreen`; version 0.2.0 → 0.3.0. No new native modules.
9. **Files changed**: 90+ (commits `d60151f`, `83b4aad`, `f35f526`, `bf28873` on master).
10. **Tests**: none added — the repo has no RN component/unit harness; `classify`/`gridColumns` are pure and test-ready when one lands (recorded gap).
11. **Manual scenarios**: §5 matrix; executable on the founder's iPad (Split View drags, rotation mid-edit, deep links at expanded, sheet keyboard flows).
12. **Known limitations**: settings/flash/travel/support master-detail deferred (caps only); dashboard/insights keep single-column layouts inside the wide cap (no 2-col restructure); SubNav section switches drop the pane selection; hardware-keyboard shortcuts and hover states not implemented; Android tablets untested on hardware; no automated regression net.
13. **Remaining recommendations**: settings hub master-detail (biggest UX win left), deposits-selects-into-requests-pane, 2-col dashboard at expanded, intrinsic-width waitlist filter tabs, anchored popover for the rail account menu, `md:`/`lg:` NativeWind screens config once cosmetic variants are wanted.
14. **New native/EAS build required**: YES for everything (no OTA); the config flips are prebuild-level.
15. **Honest assessment**: **tablet-adaptive, solidly** — intentional nav, readable widths everywhere, real master-detail on the three highest-value workflows, deterministic rotation/resize with lossless drafts. Not yet "fully tablet-optimized": the deferred items in (12) are what separates adaptive from optimized. Not a stretched phone app.

**Adversarial review (Phase 5)**: 70 agents (6 finders, 2 skeptics per finding) over the full diff. 32 findings → 29 confirmed → all fixed or explicitly accepted with rationale in commit `bf28873` (headliners: focus-gating both reconciliation rules, dismissTo vs duplicated navigators, grid column floors on phones, sheet flex-shrink chains, VoiceOver sheet shields).
