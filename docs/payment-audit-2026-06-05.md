# Payment / Deposit feature — multi-agent audit (2026-06-05)

**Scope:** the deposit-collection money feature on branch `payment-stripe` (HEAD `37ad8de`): Stripe Connect **Custom** in-app KYC onboarding (Slice 79), the deposit charge/fee/refund path, multi-currency (Slice 79d), manual fallback, and all surrounding UI/UX, emails, and docs.

**Method:** four read-only specialist agents run in parallel — Security & money-correctness, UI/UX & copy, Functionality & process (end-to-end), and Docs/vision/lost-tasks. Findings below are deduplicated and re-prioritised; the two highest-impact items were independently re-verified by the orchestrator.

**Headline verdict:** the feature is **architecturally on-vision and ~90% code-complete**, but **not launch-ready**. Two code blockers (a dead-but-dangerous duplicate actions module; unwired refund-on-cancellation), the never-run money-path verification, and an unresolved unit-economics decision stand between here and "ready & on-vision."

State at audit: `typecheck` clean, **284 tests green**, `lint` 0 errors / 6 pre-existing warnings; branch in sync with origin.

## What's confirmed solid (don't re-litigate)

- Canonical deposit path: full-3% `application_fee` under Custom (`fees.payer:application`), correct MoR posture (`on_behalf_of` + `transfer_data.destination`), idempotent create/refund (`deposit-intent-${id}`, `refund-deposit-${id}` + audit-gate), webhook amount re-validation, no client-side amount/currency tampering.
- KYC PII never persisted or logged (H-1/H-2). C-2 RLS leak genuinely closed by migrations 0030/0031. Server-only guard on `stripe-connect.ts` (M-4).
- Money-wall holds in the live path: only the explicit KYC opt-in creates a Connect account; `deriveConnectRouting`/`getConnectRoutingForArtist` gates every live money surface; no card charge before `charges_enabled`.
- Magic-link tokens: 32-byte random, SHA-256 hashed, rotated per state change, 30-day expiry. AuthZ via `getAuthorisedBooking` / `artist_id === user.id` everywhere.
- Cron overdue reminders: timezone-correct, double-send-guarded, currency-aware. Currency threads correctly through request→PI→portal→display→all 4 emails (misses only P0-6 + P1-3).
- Emails are the most polished surface: real hosted PNG logo, Stripe-secure + deposit-goes-to-artist + no-fee trust copy, clear CTA. Refund consequence copy is correct. Manual-vs-card mode signalling (F7 `hasDepositIntent`) is wired right.

## Vision alignment (one line)

The build faithfully serves Inklee's vision — calm, designer-first, free universal core, optional opt-in deposits, artist stays merchant of record, no Stripe-signup friction. The gaps are not vision drift; they are an **unverified money rail**, an **unproven unit economics**, and a **half-built refund/cancellation flow** that make it not-yet-trustworthy to put real money through.
