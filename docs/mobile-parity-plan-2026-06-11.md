## 1. Dashboard + home widgets

The mobile Home tab (apps/mobile/app/(tabs)/index.tsx) is a thin "what needs action now" screen backed by a single GET /api/mobile/home aggregate (apps/web/src/app/api/mobile/home/route.ts, typed by MobileHome in packages/shared/src/mobile-api.ts). It renders: a greeting H1 ("Hi, {displayName}" / "Home" fallback), a BooksToggle card (open/close control, optimistic), one combined stat Card showing pendingCount + waitlistCount as two numbers, a "Needs a reply" list of up to 5 pending requests as plain non-tappable Card rows (client + placement/date), and an "Upcoming" list of up to 5 approved bookings, plus pull-to-refresh. It does NOT implement the web dashboard's widget model at all: no per-widget visibility toggles, no /settings/dashboard screen, no guest-spots widget, no "Your links" (booking/waitlist share) widget, no onboarding/bio nudge cards, no analytics entry card, no "View all / Calendar / Plan / View / Edit" header links, no row deep-links to detail, no loading skeleton, and no all-hidden empty state. Books open/closed status is also surfaced glanceably in the top bar (BooksStatusPill.tsx) mirroring web. The home aggregate omits the data the web dashboard reads (guest-spot trip legs, bio, dashboard_widgets toggles, total-received-count for the zero-request convenience), and there is no /api/mobile route for dashboard widget settings.

## 2. Bookings: requests / overview + request detail

The mobile app has a real but thin slice of this surface. Bookings is a tab (apps/mobile/app/(tabs)/bookings/_layout.tsx) with a 3-chip sub-nav (Requests / Calendar / Clients) — note: the third native sub-view is Calendar, not Waitlist; Waitlist lives at a separate top-level route apps/mobile/app/waitlist.tsx. The Requests list (apps/mobile/app/(tabs)/bookings/index.tsx) is a FlatList of stacked cards (client label, placement/size/preferred-date line, StatusPill, "Deposit paid" hint) fetched from GET /api/mobile/bookings with NO status filter param, NO trip filter, NO pagination cursor wired (the keyset cursor exists server-side but the screen never sends ?status or ?cursor), and a single generic empty state with no share/copy link. The Clients roll-up (clients.tsx) is full with client-side search. The request detail (apps/mobile/app/bookings/[id].tsx) renders identity + StatusPill + "Requested {relative}", a Tattoo section (placement/size/description), Reference (link with http(s) guard + a TEXT COUNT of images, no gallery/lightbox), Schedule (preferred date only), a Deposit summary section, and the BookingActions toolbar (apps/mobile/src/components/booking/BookingActions.tsx). BookingActions covers the core FSM transitions well: Accept (direct, no popups), Request deposit (inline form with prefilled defaults, 3% fee split, manual-vs-Connect copy), Pass (2-step confirm), Mark deposit received (+ "Waiting for card payment" panel), Refund deposit, Cancel booking (with refund disclosure). StatusPill uses shared humanStatusLabel + brand colours, matching web vocabulary. Missing entirely from mobile: status/trip filter chips, collapsible Filter pill, waitlist origin tag, share-link empty state, reference image gallery/lightbox/annotations, goods/interested-in-buying, accept studio+goods confirm popup, deposit fee-breakdown is present but no Stripe test-mode banner, no Connect nudge link, no deposit reminder / reconfirmation actions, no communication activity timeline, no "show more details" disclosure, no feature-intro modal, no loading skeleton (uses null/spinner), no waitlist convert/demand-by-city/shareable link, no slot time / location / Instagram / flash-item / custom-question fields, no legacy redirect concept.

## 9. Goods: products / variants / sales
