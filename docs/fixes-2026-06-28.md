## 6. Deposit overdue stuck after a client cancels + no restart loop  [MAIN]  [MONEY PATH]
Founder report: when a card deposit is cancelled by the client, the artist's deposit entry is stuck "overdue" with no way to revoke it; should be markable cancelled/solved, and the artist should be able to send a NEW deposit request afterward so the process isn't permanently stuck.

Root cause: `depositState()` (the single source-of-truth classifier, `apps/web/src/lib/deposit-state.ts`) derived state purely from the deposit columns and was BLIND to `booking.status`. A client cancel sets the booking to `cancelled` and cancels the Stripe intent, but leaves `deposit_amount` + a past `deposit_due_at` with no `deposit_paid_at` → classified "overdue" forever. Surfaced in: the deposits overview (web `/bookings/deposits` + mobile, via `getDepositsOverview` — inflated the Outstanding/Overdue hero too), the request-detail Deposit card (stale "Due <past date>"), and the dashboard "Action required" feed / Home deposit glance (`getDashboardData`). And once terminal, web `StatusActions` + mobile `BookingActions` rendered NO actions, so there was no way out and no re-request (the FSM also blocks `deposit_pending → deposit_pending`).

Decisions (founder, 2026-06-29): (a) reopen ANY dead booking (cancelled or passed) when no deposit money was kept; (b) show dead deposits in a quiet "Cancelled" group, out of the chase totals.

Fix (16 files + 1 new route):
- `depositState()` is now status-aware: a 4th `bookingStatus` arg; an UNPAID deposit on any non-`deposit_pending` booking is classified the new terminal state `cancelled` (paid/refunded still win, so a forfeited deposit stays "paid"/collected). Callers `getDepositsOverview` + `getDashboardData` pass status; the dashboard unpaid-deposits query is now scoped `.eq("status","deposit_pending")` so dead deposits never surface as overdue or as a "Mark received" action item. `MobileDepositListItem.state` union + `DepositState` widened with `"cancelled"`.
- Quiet "Cancelled" section added to both deposits overviews (web + mobile); rank `cancelled:3` (before refunded); excluded from outstanding/overdue/collected rollups. Request-detail Deposit card shows "Cancelled" for a dead booking instead of stale "Due". Mobile booking-detail deposit status shows "Cancelled" and hides the Due-by row.
- Reopen loop: `reopenBookingCore` (`@/lib/server/bookings`) — money-gated (refuses if `deposit_paid_at` set), cancels any lingering unpaid intent, clears the dead deposit columns, flips `cancelled|rejected → pending` under a status-gated conditional UPDATE + audit row. Shared `isReopenable()` helper added to the FSM (kept OUT of the generic TRANSITIONS table so `isTerminal` semantics are unchanged everywhere else). Web server action `reopenBooking` + new mobile route `POST /api/mobile/bookings/[id]/reopen` + mobile client `reopenBooking`. UI: a "Reopen booking" action on a dead, money-free booking (web `StatusActions` via new `canReopen` prop; mobile `BookingActions`); after reopen the booking returns to pending and the full Accept / Request deposit / Pass set comes back. Timeline shows "Reopened".
